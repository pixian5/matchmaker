## **1. 已修复/改善的重点**

## **2. 依然存在的"严重"风险与架构隐患**

虽然加了锁，但由于核心的"整包读写"逻辑没变，项目依然面临以下三个核心问题：

#### **🔴 风险一：并发竞争导致的"静默数据丢失"（Lost Update）**
这是由于 `PUT /api/state` 依然是**全量覆盖**导致的。
*   **场景**：红娘 A 在 10:00 拉取了全量状态（v1）；红娘 B 在 10:01 也拉取了全量状态（v1）。
*   **冲突**：A 在 10:05 修改了"机构信息"并 `PUT` 回去；B 在 10:06 修改了"用户信息"并 `PUT` 回去。
*   **结果**：B 的写入会**彻底抹除** A 在 10:05 做出的所有修改。因为 B 提交的是自己本地的旧版本 v1，里面不包含 A 的改动。
*   **严重性**：随着红娘和管理员人数增加，这种数据冲突将变得不可接受。

#### **🟠 风险二：数据库"自杀式"更新逻辑（TRUNCATE 性能瓶颈）**
您的 `syncNormalizedState()` 函数依然保留了以下逻辑：
```js
await client.query(`truncate table deals, match_requests, ... restart identity cascade`);
```
*   **问题**：每次有人修改一个微小的资料（比如改个昵称），后端都会把 PostgreSQL 里的 7 张业务表**全部清空再重新插入一遍**。
*   **后果**：
    1.  **锁死**：`TRUNCATE` 需要最高等级的排他锁，大并发下会导致请求排队甚至超时。
    2.  **日志爆炸**：这种全量重写会产生极大的 WAL 日志，浪费磁盘 IO。
    3.  **自增 ID 归零**：由于使用了 `restart identity`，如果您有其他表引用了这些 ID 却没加级联，系统会崩溃。

#### **🟡 风险三：HMAC 令牌的安全局限**
您目前使用的是自实现的 HMAC Token [GitHub](https://github.com/pixian5/mediapeople/blob/master/readme.md)。
*   **风险**：如果令牌没有包含过期时间（Exp）或服务端没有黑名单机制，一旦泄露，该令牌将永久有效。
*   **建议**：迁移到标准的 JWT（JSON Web Token），利用其自带的 `exp` 声明来强制令牌过期。

## **3. 针对性的重构建议（P2 阶段）**

您在 `readme` 中已经规划了"拆分角色 API"，这非常正确。为了根治上述问题，建议按以下顺序操作：

### **第一步：引入"乐观锁"版本号（解决并发冲突）**
在不彻底拆分接口前，这是成本最低的防冲突方案：
1.  在 `app_state` 表增加 `version` 字段。
2.  `GET /api/state` 返回当前 `version`。
3.  `PUT /api/state` 时，前端必须带上 `version`。
4.  后端执行 `UPDATE ... WHERE id=1 AND version=$expected_version`。如果更新行数为 0，说明别人改过了，返回 `409 Conflict`。

### **第二步：废弃 TRUNCATE，改用 UPSERT（解决性能瓶颈）**

*   使用 `INSERT ... ON CONFLICT (id) DO UPDATE` 语法。
*   对于已删除的数据，对比前端传来的 ID 列表，删除数据库中不在列表里的行。
